# PRD Analyst Skill

1-pager를 토대로 실질적인 구현이 가능하도록 화면 명세의 토대인 기능 명세, 성공 지표, 예외 비즈니스 정책 등을 상세히 수립하는 **제품 요구사항 명세(PRD)** 스킬입니다.

## PRD 작성 가이드라인 및 템플릿

작성 시 다음 명세 구조를 엄격히 준수하여 항목을 기재합니다:

```markdown
# [{YY.MM.DD}] {프로젝트명} ({팀/조직명})
> 작성일: {YYYY-MM-DD} / 작성자: {이름} / 상태: Draft / 버전: v0.1

## 0. 문서 개요
### 0-1. 프로젝트 요약
- {아이디어 한 줄 정리}
### 0-2. 배경
- {현재 직면한 상황 및 VOC 분석}

## 1. 목표 (Goal)
### 1-1. 프로젝트 목표
- **사용자 목표**: {고객에게 제공할 실질 가치}
- **비즈니스 목표**: {회사 비즈니스에 기여할 목표}
### 1-2. 성공 기준
| 지표 | 현재 | 목표 |
| :--- | :--- | :--- |
| {예시 지표} | {현재 수치} | {타겟 수치} |

## 2. KPI 및 측정지표
### 2-1. 핵심 KPI
| KPI명 | 정의 | 계산식 | 목표 | 측정 주기 | 측정 도구 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| {KPI} | {지표 정의} | {수식} | {목표} | {주기} | {도구} |
### 2-2. 보조 지표
- {보조 지표 목록}
### 2-3. 모니터링 지표
- {유입, 탐색, 퍼널, 리텐션 등의 정성/정량 측정 요소}

## 3. 사용자 및 이해관계자
### 3-1. 대상 사용자 + 제외 대상
- **대상**: {타겟 세그먼트}
- **제외**: {예외 세그먼트}
### 3-2. 이해관계자
| 조직 | 역할 |
| :--- | :--- |
| {부서명} | {협업 요건} |

## 4. 정책 정의
### 4-1. 기본 정책
- {이용 조건, 권한 정책 등}
### 4-2. 상태값 정의
| 상태 | 설명 |
| :--- | :--- |
| {예시: INIT} | {초기 상태} |
### 4-3. 예외 정책
| 상황 | 처리 |
| :--- | :--- |
| {API 타임아웃} | {오류 화면 노출 및 3회 재시도 폴백} |

## 5. 서비스 구조
### 5-1. 서비스 흐름 (텍스트 다이어그램)
- {흐름을 간단한 텍스트 또는 화살표로 묘사}
### 5-2. 유저 플로우
- {유저가 겪는 단계별 핵심 행동 순서}

## 6. 상세 요구사항
### 6-1. 기능 요구사항
| ID | 기능 | 설명 | 우선순위 |
| :--- | :--- | :--- | :--- |
| **REQ-01** | {기능명} | {구체적인 상세 동작 정책} | [P0/P1/P2] |
### 6-2. 운영 요구사항
- {어드민 제어 및 알림 정책}
### 6-3. 데이터 요구사항
| 항목 | 설명 |
| :--- | :--- |
| {테이블명/필드명} | {기록 보존 정책} |

## 7. UX/UI 고려사항
- {로딩, 실패, 빈 화면 처리 원칙}

## 8. 기술 고려사항
- {인증, 보안 정책, 동시성 처리 등}

## 9. 실험 및 검증
- {A/B 테스트 상세 가이드 및 대안 가설}

## 10. 리스크 및 대응
| 리스크 | 대응 |
| :--- | :--- |
| {인프라 부하} | {레이트 리미터 적용} |

## 11. 일정 및 운영계획
| 단계 | 기간 |
| :--- | :--- |
| {기획} | {기간} |

## Appendix
- {용어 정의 및 히스토리 기록}
```

## 핵심 원칙
- **예외 처리 필수**: 결제 에러, 네트워크 끊김, 다중 중복 클릭 등 서비스 안정성에 직결되는 엣지 케이스 정책을 반드시 포함합니다.
- **애매성 제거**: '~할 수 있다' 같은 허용적 표현을 배제하고, '~한다' 혹은 '~해야 한다' 등의 명확한 제약식으로 기술합니다.
