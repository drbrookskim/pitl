# [24.05.20] 정보 과부하 해결을 위한 온디맨드 뉴스 큐레이션 플랫폼 'Briefly' (Product Team)

## 1. 프로젝트 한 줄 요약
사용자가 지정한 관심 키워드와 주제에 맞춰 구글 뉴스 API를 통해 실시간으로 맞춤형 뉴스를 수집하고, AI 요약 기술을 더해 개인화된 형태로 배달하는 온디맨드 뉴스 큐레이션 서비스.

## 2. 대상
| 대상 | 상황/특징 |
| :--- | :--- |
| **전문직 및 비즈니스 종사자** | 기획자, 마케터, 투자자 등 특정 산업 트렌드 및 경쟁사 동향을 매일 모니터링해야 하나, 검색에 너무 많은 시간을 소모하는 직군 |
| **대학생 및 연구원** | 특정 학술 주제, 기술 트렌드, 시사 이슈에 대한 깊이 있는 뉴스를 선별적으로 수집하고 아카이빙하고자 하는 사용자 |
| **정보 피로감을 느끼는 대중** | 포털 사이트의 자극적인 황색 언론, 중복 기사, 광고성 기사에 지쳐 신뢰할 수 있는 뉴스만 골라 읽고 싶어 하는 사용자 |

## 3. 문제 정의

### 고객 문제 (Customer Problem)
* **정보 과부하(Information Overload):** 포털 뉴스는 광고와 중복 기사로 가득 차 있어, 정작 필요한 정보를 필터링하고 핵심을 파악하는 데 일평균 40분 이상의 과도한 시간이 소모됨.
* **개인화 부재:** 기존 뉴스 앱은 대중적인 실시간 검색어 위주로 피드를 구성하여, 개인의 마이너하거나 전문적인 관심사를 충족시키지 못함.

### 비즈니스 문제 (Business Problem)
* **낮은 리텐션 및 체류 시간:** 기존 뉴스 서비스들은 차별화된 가치 제안(Value Proposition) 부족으로 인해 사용자 이탈률이 높고, 유료 구독 모델로의 전환율이 극히 저조함(평균 1% 미만).

### 데이터 기반 현황 테이블
| 지표 | 현재 수치 (업계 평균) | 문제점 |
| :--- | :--- | :--- |
| **일평균 뉴스 검색 시간** | 42분 | 불필요한 중복 기사 및 광고 노출로 인한 시간 낭비 심화 |
| **뉴스 앱 이탈률 (Bounce Rate)** | 68% | 개인화되지 않은 일방향적 뉴스 피드로 인한 흥미 상실 |
| **유료 구독 전환율 (Conversion)** | 1.2% | 무료 포털 뉴스와의 차별성 부족으로 지불 용의성(WTP) 저하 |

### Root Cause (근본 원인)
기존 뉴스 플랫폼이 '공급자 중심'의 대량 송출 방식을 고수하고 있으며, 사용자의 '맥락(Context)과 실시간 니즈'를 반영하여 정보를 정제해 주는 온디맨드 필터링 및 요약 기술이 부재함.

## 4. 목표
| 항목 | 목표 수치 | 비고 |
| :--- | :--- | :--- |
| **뉴스 탐색 시간 단축** | 일평균 42분 -> **10분 이하 (75% 감소)** | 핵심 요약 및 필터링을 통한 효율 극대화 |
| **서비스 리텐션 (D+7)** | 현재 15% -> **45% 이상 달성** | 개인 맞춤형 알림을 통한 재방문 유도 |
| **유료 구독 전환율** | 현재 1.2% -> **5.0% 달성** | AI 요약 및 외부 채널(슬랙/메일) 연동 유료화 |
| **API 호출 최적화** | API 비용 **30% 절감** | Redis 캐싱 알고리즘 도입을 통한 중복 호출 방지 |

## 5. 현황 및 분석
* **고객 VOC 인용:** 
  * *"매일 아침 IT 트렌드 뉴스를 검색하는데, 광고성 기사와 동일한 내용의 복사 붙여넣기 기사가 너무 많아 피로합니다." (30대 IT 기획자 A씨)*
  * *"내가 설정한 특정 키워드에 대한 뉴스만 깔끔하게 요약해서 매일 아침 슬랙이나 이메일로 받아볼 수 있다면 기꺼이 돈을 지불할 의향이 있습니다." (20대 스타트업 창업자 B씨)*
* **시장 트렌드:** 글로벌 뉴스 큐레이션 시장은 연평균 12% 성장 중이며, 단순 나열식 뉴스 제공에서 AI 기반 요약 및 개인 맞춤형 배달(On-Demand Delivery) 형태로 패러다임이 빠르게 전환되고 있음.

## 6. 해결방안
* **Phase 1 (MVP - 검증 단계):** 구글 뉴스 API를 연동하여 사용자가 입력한 키워드 기반의 실시간 뉴스 피드 구축. 키워드 등록 및 기본 웹 뷰어 기능 제공.
* **Phase 2 (AI 고도화 단계):** LLM(Large Language Model)을 활용한 뉴스 3줄 요약 기능 도입. 동일 이슈에 대한 중복 기사를 하나로 묶어주는 클러스터링 알고리즘 적용.
* **Phase 3 (플랫폼 확장 단계):** 온디맨드 알림 서비스(슬랙, 카카오톡, 이메일 연동) 출시. 개인화 추천 엔진 고도화 및 유료 구독 모델(Premium Plan) 도입.

## 7. 리스크
| 영향도 | 리스크 | 대응 방안 |
| :--- | :--- | :--- |
| **상** | 구글 뉴스 API 호출 제한 및 비용 상승 | Redis 캐싱 레이어를 도입하여 동일 키워드에 대한 중복 호출을 방지하고, 유료 플랜 사용자 중심으로 API 할당량을 최적화함. |
| **중** | 뉴스 저작권 및 무단 전재 이슈 | 기사 원문 링크 아웃(Outlink) 방식을 유지하며, AI 요약본은 저작권 가이드라인(공정 이용 범위)을 준수하여 원문의 단순 요약 형태로만 제공함. |
| **중** | 실시간성 저하 및 데이터 정합성 문제 | 중요 키워드에 대한 배치(Batch) 수집 주기를 단축하고, 실시간 웹소켓(Websocket) 알림 연동을 통해 데이터 딜레이를 최소화함. |

## 8. 중요도 및 긴급도
* **중요도: 상**
  * *이유:* 정보 과부하 시대에 '나만을 위한 정제된 정보'에 대한 지불 용의(Willingness to Pay)가 급증하고 있으며, 이는 플랫폼의 강력한 락인(Lock-in) 효과와 고부가가치 B2B SaaS 모델로의 확장성을 담보하기 때문임.
* **긴급도: 중**
  * *이유:* 현재 시장에 다양한 뉴스레터 서비스가 존재하나, 실시간 '온디맨드' 및 'API 기반 자동화 큐레이션'을 완벽히 구현한 서비스는 초기 단계임. 시장 선점을 위해 3개월 내 MVP를 출시하여 시장 반응을 검증해야 함.
