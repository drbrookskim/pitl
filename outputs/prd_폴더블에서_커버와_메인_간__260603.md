# [24.10.24] 폴더블 폼팩터 특화 UX 'FlexFlow' 프로젝트 상세 요구사항 명세서

## 0. 문서 개요
본 문서는 폴더블 기기의 물리적 상태(힌지 각도, 거치 방향)를 감지하여 사용자에게 최적화된 기능적 가치를 제공하는 'FlexFlow' 프로젝트의 상세 요구사항을 정의한다. 단순 화면 연속성을 넘어, 폼팩터의 물리적 변화를 능동적인 기능 트리거로 활용하여 폴더블 기기의 독점적 가치를 창출하는 것을 목적으로 한다.

## 1. 목표 (Goal)
* **프로젝트 목표:** 폴더블 폼팩터의 물리적 상태를 인지하는 '액티브 펑션(Active Function)' 도입을 통한 사용자 경험 혁신 및 브랜드 락인(Lock-in) 강화.
* **성공 기준:**
| 구분 | 목표 수치 | 비고 |
| :--- | :--- | :--- |
| Flex 모드 일평균 사용 시간 | 15.0분 | 현재 대비 약 3.5배 증대 |
| 폴더블 특화 기능 주 1회 이상 사용률 | 35.0% | 신규 기능 활성화 유도 |
| FlexFlow Retention (D+7) | 45.0% 이상 | 사용자 잔존율 극대화 |

## 2. KPI 및 측정지표
* **핵심 KPI:** Flex 모드 일평균 사용 시간, 특화 기능 활성화율, D+7 리텐션.
* **보조 지표:** 커버-메인 화면 전환 만족도, 폼팩터 기능 잠금(Lock) 사용 빈도.
* **모니터링 지표:** 센서 오작동(의도치 않은 전환) 발생률, 배터리 소모량 변화 추이.

## 3. 사용자 및 이해관계자
* **대상 사용자:** 폴더블 헤비 유저, 생산성 중시 비즈니스 프로페셔널, 모바일 크리에이터.
* **제외 대상:** 폴더블 기능을 단순 미디어 시청용으로만 사용하는 라이트 유저(기능 강제 노출 최소화).
* **이해관계자:** OS 개발팀(센서 API), UX/UI 디자인팀, 서드파티 파트너십팀, QA팀.

## 4. 정책 정의
* **상태값 정의:**
| 상태 | 정의 | 감지 조건 |
| :--- | :--- | :--- |
| 닫힘 | 0도 | 힌지 센서 0도 |
| 미세 접힘 | 1~45도 | 힌지 센서 1~45도 |
| 직각 거치 | 70~110도 | 힌지 센서 70~110도 + 가속도 센서(거치 판별) |
| 완만 거치 | 111~170도 | 힌지 센서 111~170도 |
| 펼침 | 171~180도 | 힌지 센서 171~180도 |

* **예외 정책:**
  * **센서 오작동:** 상태 변화 후 1.5초간 유지 시에만 기능 전환(Debounce).
  * **사용자 거부:** 퀵패널 내 '폼팩터 기능 잠금' 활성화 시 자동 전환 중단.

## 5. 서비스 구조 (유저 플로우)
1. **상태 감지:** 센서 허브가 힌지 각도 및 기기 방향 실시간 모니터링.
2. **컨텍스트 매칭:** 현재 실행 중인 앱과 상태값(예: 90도+카메라) 매칭.
3. **UI/UX 전환:** 시스템이 앱에 상태값 전달 → 앱 내 특화 UI(Flex-Camera 등) 자동 렌더링.
4. **사용자 피드백:** 전환 시 미세한 햅틱 피드백 제공.

## 6. 상세 요구사항
| ID | 요구사항명 | 우선순위 | 상세 내용 |
| :--- | :--- | :--- | :--- |
| REQ-01 | 폼팩터 상태 감지 엔진 | P0 | 힌지/자이로 센서 기반 5단계 상태 판별 API 개발 |
| REQ-02 | Flex-Camera 모드 | P1 | 90도 거치 시 상단 뷰파인더/하단 편집기 분할 UI 제공 |
| REQ-03 | Flex-Office 모드 | P1 | 텐트 모드 시 전면 타이머/후면 스크립트 분할 표시 |
| REQ-04 | 디바운스 알고리즘 | P0 | 1.5초 유지 조건 적용으로 오작동 방지 |
| REQ-05 | 폼팩터 기능 잠금 | P2 | 퀵패널 내 토글 버튼 제공 (사용자 제어권 보장) |

## 7. UX/UI 고려사항
* **로딩:** 상태 전환 시 0.3초 이내 UI 전환 완료(애니메이션 최적화).
* **실패:** 센서 데이터 불확실 시 기존 UI 유지 및 사용자에게 '기능 전환 불가' 알림(토스트 메시지).
* **빈 화면:** 서드파티 앱 미지원 시 기본 레이아웃 유지 및 'Flex 모드 미지원' 안내 문구 노출.

## 8. 기술 고려사항
* **저전력 설계:** 메인 AP 개입 최소화, 센서 허브(Sensor Hub)에서 1차 연산 처리.
* **보안:** 센서 데이터는 시스템 레벨에서만 처리하며, 서드파티 앱에는 추상화된 상태값(Enum)만 전달.

## 9. 실험 및 검증
* **A/B 테스트:** FlexFlow 기능 자동 활성화 그룹 vs 수동 활성화 그룹 간 리텐션 비교.
* **사용성 테스트:** 20~40대 타겟 그룹 대상 힌지 각도별 기능 전환 만족도 정성 조사.

## 10. 리스크 및 대응
* **리스크:** 센서 오작동으로 인한 피로도 증가.
  * **대응:** 1.5초 디바운스 적용 및 기능 잠금 버튼 제공.
* **리스크:** 서드파티 앱 미지원.
  * **대응:** 주요 10대 앱 대상 커스텀 매핑 프로필 배포 및 SDK 가이드라인 제공.
