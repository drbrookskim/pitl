# [26.05.31] 유동적 상품기획 자동화 플랫폼 (PITL-Process) (Product-Planning Team)
> 작성일: 2026-05-31 / 작성자: Product-Planning Team / 상태: Draft / 버전: v0.1

## 0. 문서 개요
### 0-1. 프로젝트 요약
- 텍스트 기반 상품 컨셉 아이디어를 입력받아 비즈니스 1-pager, 기능 사양 PRD, 그리고 CTA와 정보 우선순위가 확연히 다른 3가지 시안의 ASCII 와이어프레임을 유기적 삼각 정합성 검증 루틴을 통과시켜 디스크에 안전하게 배포하는 지능형 기획 프로세스 플랫폼입니다.
### 0-2. 배경
- 기획서의 비즈니스적 당위성(1-pager)과 구현 요건(PRD), 화면 구성(UX)이 상호 매칭되지 않아 발생하는 기획 결함 및 재작업 병목을 줄이기 위해, 삼각 교차 검증을 자동 수행하는 AI 팀 하네스가 요구됩니다.

## 1. 목표 (Goal)
### 1-1. 프로젝트 목표
- **사용자 목표**: 기획에 대한 복잡한 고민 없이 검증된 템플릿과 정교한 정책 설계가 탑재된 마크다운 문서 3종을 실시간으로 획득하여 설계 착수 시간을 대폭 단축합니다.
- **비즈니스 목표**: 초기 요구사항 정의 및 화면 스케치 합의를 1회 워크플로우로 단번에 완료하여 스프린트 준비 공수를 80% 절감합니다.
### 1-2. 성공 기준
| 지표 | 현재 | 목표 |
| :--- | :--- | :--- |
| **기획 초안 통과율** | 61.8% (첫 회의 합의) | **90% 이상** (재회의 없는 즉각 승인) |
| **문서 정합성 오류 건수** | 평균 기획서당 4.2건 (스펙 불일치) | **0건** (교차 삼각 정합성 검수로 사전 차단) |

### 2-2. 보조 지표
- **에이전트 수정 루프 회수**: Reviewer가 반려하여 Writer가 재작성한 평균 사이클 회수 (목표: 1.2회 이하).
- **PDF 변환 통과율**: 마크다운 파일의 PDF 컴파일 완료 비율 (목표: 98% 이상).

### 2-3. 모니터링 지표
- **유입 지표**: 아이디어 입력창 노출 및 제출 버튼 클릭 수.
- **탐색 지표**: 생성 완료 후 1-pager, PRD, 와이어프레임 3안 간의 탭 전환 빈도 및 스크롤 체류 시간.
- **퍼널 지표**: 아이디어 입력 → 1-pager 완성 → PRD 완성 → 와이어프레임 완성 → 최종 마크다운/PDF 다운로드로 이어지는 퍼널 완료율 (목표: 80% 이상).
- **리텐션 지표**: 기획자가 주 2회 이상 플랫폼에 복귀하여 신규 기획 프로세스를 실행하는 비율.

## 4. 정책 정의
### 4-1. 기본 정책
- **기획 템플릿 표준화**: 모든 상품기획 프로세스는 1-pager(비즈니스), PRD(상세 정책), 와이어프레임 3안(UX 레이아웃)의 3대 패키지 문서를 일괄 생성해야 합니다.
- **사용자 언어 감지**: 사용자가 한국어로 입력하면 최종 문서 3종 모두 고품격 한국어로 생성하며, 영어로 입력 시 영어로 생성합니다.
### 4-2. 상태값 정의
| 상태 | 설명 |
| :--- | :--- |
| **INIT** | 사용자의 기획 주제를 입력받고 모드(에세이 vs 상품기획) 판별을 진행하는 상태. |
| **PLANNING** | 1-pager 요약서 초안 및 비즈니스 정량 목표를 기획하는 상태. |
| **PRD_DRAFTING** | 비즈니스 목표를 개발 요건 REQ ID 및 정책으로 정교화 명세하는 상태. |
| **UX_DESIGNING** | PRD 요건에 매핑되는 3가지 ASCII 와이어프레임 시안(Type A/B/C)을 작조하는 상태. |
| **QA_REVIEW** | 1-pager, PRD, 와이어프레임 간 삼각 교차 검증을 수행하고 보고서를 발행하는 상태. |
| **COMPLETED** | 검수를 통과한 3종 문서에 보안 필터링을 적용하여 디스크에 배포 완료한 상태. |
### 4-3. 예외 정책
| 상황 | 처리 |
| :--- | :--- |
| **QA 검수 불일치 감지** | Reviewer가 스펙 누락 또는 목표 불일치 감지 시 즉시 상태를 해당 단계(`PLANNING` 또는 `UX_DESIGNING`)로 롤백하고 구체적 보완안을 동봉하여 `SendMessage`로 반려 및 재작업을 지시합니다 (최대 2회). |
| **수정 루프 2회 초과 시** | 에이전트 간의 수정 루프가 2회를 초과할 경우 지연 방지를 위해 Reviewer가 직권으로 누락 요구사항을 diff 문장으로 합산 보완하여 `COMPLETED` 상태로 강제 전환합니다. |
| **파일명 쉘 주입 공격 시** | 파일명에 경로 이탈 문자(`..`, `/`)나 쉘 메타문자(`;`, `&`, `*` 등)가 감지될 경우, 오케스트레이터가 이를 언더바(`_`) 또는 하이픈(`-`)으로 강제 치환 및 필터링(Sanitize)하여 보안 배포합니다. |

## 5. 서비스 구조
### 5-1. 서비스 흐름 (텍스트 다이어그램)
```
[사용자 아이디어 입력] 
   │
   ▼
[Mode Detection] ──→ (Product Mode 판별)
   │
   ▼
[TeamCreate: 4인 에이전트 팀 스폰 (strategist, analyst, architect, reviewer)]
   │
   ▼
[Task 1: product-strategist 1-pager 기획] ──→ _workspace/01_strategist_1pager.md
   │
   ▼ (SendMessage)
[Task 2: prd-analyst PRD 요건 명세] ──→ _workspace/02_prd_spec.md
   │
   ▼ (SendMessage)
[Task 3: ux-architect 와이어프레임 3안 작조] ──→ _workspace/03_ux_wireframes.md
   │
   ▼ (SendMessage)
[Task 4: product-reviewer 삼각 교차 QA 검수] ──→ [PASS] 판정 및 04_reviewer_audit.md 발행
   │
   ▼
[파일명 필터링 및 디스크 배포] ──→ outputs/ 기획 서류 3종 적재 ──→ [최종 완료 보고]
```

### 5-2. 유저 플로우
1. 사용자가 CLI 또는 웹 화면 상에서 제품 아이디어 텍스트를 제출합니다.
2. 시스템이 기획 4인 에이전트 팀과 4종의 태스크 의존성을 선언하고 백그라운드 오케스트레이션을 가동합니다.
3. 1-pager 요약 → PRD 명세 → ASCII 와이어프레임 3안 제도 단계가 순차적으로 전개되며, 각 결과물은 `_workspace/` 경로에 intermediate 파일로 기록됩니다.
4. 검수 에이전트가 삼각 일치성을 전수 진단하고 `[PASS]` 판정을 획득하면, 파일명을 안전하게 정제하여 `outputs/` 아래에 마크다운 파일로 영구 배포합니다.
5. 사용자는 탭 뷰어를 통해 1-pager, PRD, 와이어프레임을 우아한 스타일로 탐색 및 복사합니다.

## 7. UX/UI 고려사항
- **로딩 상태**: 에이전트 팀이 기획 및 검수를 전개하는 동안, 타임라인 인터랙티브 인디케이터(Phase 1: 기획 요약 중... → Phase 2: 명세 분해 중... → Phase 3: 와이어프레임 제도 중...)를 실시간 노출합니다.
- **실패 상태**: 검수가 2회 이상 실패하거나 에이전트가 유휴 상태에 직면해 타임아웃(30초)이 날 경우, 붉은색 경고 패널과 함께 현재까지 작성된 마크다운 임시 데이터와 복구 가이드를 제공합니다.
- **빈 상태**: 생성된 기획서가 없을 경우 대시보드에 우아한 일러스트와 함께 "새로운 아이디어를 입력하여 유동적 기획 프로세스를 동작시키세요" 가이드를 띄웁니다.

## 8. 기술 고려사항
- **에이전트 모델 제약**: 하네스의 철학적 수준과 완성도 높은 ASCII 아트 출력을 보장하기 위해, 모든 서브 에이전트 호출에는 `model: "opus"`를 고정 파라미터로 명시하여 성능 하락을 방지합니다.
- **보안 격리**: 파일 쓰기는 사전에 협의된 `pitl-planning-in-the-loop/outputs/` 및 임시 작업 경로 `_workspace/` 내부로 한정하며 외부 시스템 경로 작성을 원천 금지합니다.

## 9. 실험 및 검증
- **A/B 테스트 계획**: AI 생성 기획서의 필드 실적 향상을 위해, "기존 수동 작성 기획서(A안)" 대비 "본 플랫폼 자동화 기획서(B안)"를 실제 개발 스프린트에 도입하여, 개발자 및 디자이너 피드백 만족도 점수(CSAT) 및 스프린트 지연 일수를 측정 및 교차 분석합니다.
